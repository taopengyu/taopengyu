InnoDB Page Structure

操作系统：

从磁盘存取数据时以页（逻辑单位，1页=4kb）为基本单位

依据局部性原理，大部分指令在局部区域存在联系：

假设指令1从0x01存取1kb，就从磁盘中取出1页（4kb）的数据放在内存中，那么指令2从0x02 存取1kb，无需读取磁盘，直接使用内存数据。

InnoDb:

InnoDB stores all records inside a fixed-size unit which is commonly called a “page” (though InnoDB sometimes calls it a “block” instead). Currently all pages are the same size, 16KB.

——————https://dev.mysql.com/doc/internals/en/innodb-page-structure.html

在InnoDb中，1页=16kb

主键索引与页目录

假设一条数据为4b，那么一页总共有4k条数据，在主键索引中，主键是排好序的，但是在查询一些比较大的主键时仍需遍历很久（链表查询很慢，插入较快），因此引入页目录可以快速检索。

The Page Directory part of a page has a variable number of record pointers. Sometimes the record pointers are called “slots” or “directory slots”. Unlike other DBMSs, InnoDB does not have a slot for every record in the page. Instead it keeps a sparse directory. In a fullish page, there will be one slot for every six records.

The slots track the records’ logical order (the order by key rather than the order by placement on the heap). Therefore, if the records are 'A''B''F''D' the slots will be (pointer to 'A') (pointer to 'B') (pointer to 'D') (pointer to 'F'). Because the slots are in key order, and each slot has a fixed size, it’s easy to do a binary search of the records on the page via the slots.

———-https://dev.mysql.com/doc/internals/en/innodb-page-directory.html

每个页目录一般包含6条数据，每个页目录的值为其包含的数据的主键最小值（下图示例为一条页目录包含两条数据）。例如我们现在要查询一条主键为3的数据（select * from t1 where id = 3），首先进入页目录为1的数据区域查找，在该区域未查到数据因此返回空，因为页目录为4的数据区域最小id为4，更不可能存在主键为3的数据。

如果我们要查的数据不在第一页会怎么办呢，例如（select * from t1 where id = 7），它会先查询第一页的数据，发现没有目标时顺着指针查询第二页，直至找到目标。

但是如果我们的页也有很多，也就相当于这个链表也很长，那么就会出现和刚才一样的问题，仍然难以查询。仍然用空间换取时间的做法，再开辟一条目录页，如下图。这样的话在执行查询命令时首先查询目录页，再按指针查询数据页，这样就形成了一颗B+树。这样的索引叫做主键索引或是聚集索引（索引和数据在一起）。

主键索引是一定要有的，如果表中没有主键，InnoDb会自动生成一条rowid（用户无法看到），剩下的索引都可以算作辅助索引，唯一索引可以当着唯一约束。

联合索引

为什么要使用联合索引

  • 减少开销。建一个联合索引(col1,col2,col3)，实际相当于建了(col1),(col1,col2),(col1,col2,col3)三个索引。每多一个索引，都会增加写操作的开销和磁盘空间的开销。对于大量数据的表，使用联合索引会大大的减少开销！
  • 覆盖索引。对联合索引(col1,col2,col3)，如果有如下的sql: select col1,col2,col3 from test where col1=1 and col2=2。那么MySQL可以直接通过遍历索引取得数据，而无需回表，这减少了很多的随机io操作。减少io操作，特别的随机io其实是dba主要的优化策略。所以，在真正的实际应用中，覆盖索引是主要的提升性能的优化手段之一。
  • 效率高。索引列越多，通过索引筛选出的数据越少。有1000W条数据的表，有如下sql:select from table where col1=1 and col2=2 and col3=3,假设假设每个条件可以筛选出10%的数据，如果只有单值索引，那么通过该索引能筛选出1000W10%=100w条数据，然后再回表从100w条数据中找到符合col2=2 and col3= 3的数据，然后再排序，再分页；如果是联合索引，通过索引筛选出1000w10% 10% *10%=1w，效率提升可想而知！
